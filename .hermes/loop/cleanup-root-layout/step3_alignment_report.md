## 3. 偏差分析

### 偏差 1: 时间估算不一致 ⚠️

| 来源 | 估算 |
|------|------|
| Spec §9 | ~50 分钟 |
| Tasks 汇总 | ~4 天（0.5+0.5+0.5+1+0.5+0.25+0.5+0.25） |

**分析:** Spec 的 50 分钟是纯文件操作时间，未计入测试适配（T4）、回归验证（T6）、外部 smoke（T7）。Tasks 的 4 天是保守估算，包含验证和修复时间。

**结论:** 以 Tasks 估算为准。实际执行中 T0-T5 核心操作可能 2-3 小时完成，T6-T7 验证约 30 分钟。总工时约 0.5 天。

### 偏差 2: Phase 执行顺序不同 ⚠️

| 来源 | 顺序 |
|------|------|
| Spec Phase 1→2 | 先删 shim（1.4）→ 再精简 headroom.go（Phase 2） |
| Plan/Tasks 依赖图 | 先精简 headroom.go（T3）→ 再删 shim（T5） |

**分析:** Plan/Tasks 的顺序更安全——确保 exports.go + compat.go + 精简 headroom.go 全部就绪后再删除旧文件，避免中间状态缺符号。

**结论:** 采用 Plan/Tasks 顺序（T3 → T4 → T5）。

### 偏差 3: 测试处理粒度不同 ⚠️

| 来源 | 描述 |
|------|------|
| Spec §3.5 | "测试文件不变" |
| Plan §5.3 | 识别 4 处测试私有符号依赖需处理 |
| Tasks T4 | 专门任务处理测试适配 |

**分析:** Spec 过于乐观。实际 `spec_a_e2e_test.go`、`spec_b_e2e_test.go`、`spec_d_coverage_test.go` 引用了 `Pipeline`、`jsonMinifierTransform`、`lineIndent`、`getPackageCCR` 等私有符号，Pipeline 去重后这些符号可能消失或移动。

**结论:** Plan/Tasks 的处理方案正确。T4 专门处理。

### 偏差 4: doc.go 处理 ⚠️

| 来源 | 描述 |
|------|------|
| Spec §3.1 | doc.go 作为独立步骤 |
| Plan §2.9 | "doc.go 为可选项，不作为必须交付" |
| Tasks | 未单独 task |

**结论:** 一致——doc.go 是可选的。如需创建，在 T1 或 T3 中顺手完成。
